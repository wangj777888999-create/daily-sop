# 智能工作台 — 全项目代码审查报告

> **审查日期**：2026-05-12  
> **审查范围**：前端 `src/`、后端 `backend/`、构建配置、数据文件、前后端契约  
> **审查方式**：三路并行 Agent（前端 / 后端 / 配置&契约）  
> **综合评分**：前端 3.2 / 5 · 后端 2.5 / 5

---

## 一、必须立刻修复（高优先级，影响运行）

### H1 · `fcntl` POSIX-only — Windows 直接崩溃

- **位置**：`backend/sops/storage.py`、`backend/artifacts.py`
- **问题**：`import fcntl` 在 Windows 上进程启动时抛出 `ModuleNotFoundError`，项目在当前 Windows 11 环境无法运行。
- **修法**：用 `filelock` 库替换，或用条件导入 `try: import fcntl / except ImportError: fcntl = None`。

---

### H2 · `output_file` 下载链路断裂 — 下载必定 404

- **位置**：`backend/sops/sandbox.py`、`backend/api/routes.py:374`
- **问题**：`SandboxExecutor.execute()` 返回字典中从未设置 `output_file`，`ExecutionLog.output_file` 始终为 `None`，下载接口每次 404。
- **修法**：在沙箱执行后解析 stdout 中 `to_excel`/`to_csv` 的文件路径并写回 `output_file`，或在生成代码末尾注入 `print("__SOP_RESULT__:" + json.dumps({"output_file": "..."}))`。

---

### H3 · `.gitignore` 遗漏 `data/uploads/` — 用户数据泄露

- **位置**：`.gitignore`
- **问题**：用户上传的 Excel/CSV 文件会被 git 追踪并可能提交到仓库。
- **修法**：添加 `data/uploads/` 和 `data/artifacts/`。

---

### H4 · 沙箱 import 过滤用正则可绕过

- **位置**：`backend/sops/sandbox.py:116-182`
- **问题**：`import   os`（多空格）、`import\nos`（换行）均能绕过正则检查。
- **修法**：改为 AST 解析白名单校验：`ast.parse(code)` + 遍历 `ast.Import` / `ast.ImportFrom` 节点，对比白名单模块列表。

---

### ~~H5 · SOP 路由定义缺失~~ — **不适用（已废弃）**

> SOP 功能已于 2026-05-12 被工具箱替代，`/sop` 路由为有意移除，无需补全。`backend/sops/` 模块保留为参考代码。H2/H4 涉及 `sops/` 的问题也随 SOP 废弃而降级为低优先级。

---

### H6 · 裸 `except:` 捕获所有异常

- **位置**：`backend/api/routes.py:55`
- **问题**：`except: params = {}` 会吞掉 `KeyboardInterrupt`、`SystemExit` 等系统异常，掩盖真实错误。
- **修法**：改为 `except (json.JSONDecodeError, KeyError, ValueError)`。

---

### H7 · 执行状态初始化不一致

- **位置**：`backend/api/routes.py:299` vs `src/services/sopApi.ts:6`
- **问题**：后端 `/execute/{sop_id}` 同步阻塞完成，初始响应返回 `"pending"`，前端永远看不到 `"running"` 状态，轮询逻辑是死代码。
- **修法**：若保持同步，直接返回最终状态；若改异步，先返回 `"running"` 再轮询。

## 二、应尽快修复（中优先级，影响功能或可维护性）

### M1 · `_validate_condition()` 不支持 `in` / `not in`

- **位置**：`backend/sops/code_generator.py:333`
- **问题**：正则白名单缺少 `in`/`not in`/`isna()` 关键字，常见过滤条件被拒绝。
- **修法**：用 `ast.parse(condition, mode='eval')` + 节点白名单替代正则。

---

### M2 · `adaptExecutionResponse()` 无法处理两种响应格式

- **位置**：`src/services/sopApi.ts:20-30`
- **问题**：`/execute` 返回 `{execution_id, status}`，`/execute/{id}/status` 返回完整结构，适配器逻辑不统一，`sop_id` 在初始响应后为 `undefined`。
- **修法**：统一后端两个接口的响应结构，或在适配器中显式处理两种 schema。

---

### M3 · `any` 滥用导致类型安全漏洞

- **位置**：`src/services/sopApi.ts:72-79`、`src/ui/pages/sop/SOPCreate.vue:77`
- **问题**：步骤映射 `data.steps.map((s: any, idx) => ...)` 和 `result as any` 绕过 TypeScript 类型检查。
- **修法**：定义 `interface BackendSOPStep { step?: number; action: string; params?: Record<string, unknown> }`。

---

### M4 · `SemanticInferencer` 导入后从未调用

- **位置**：`backend/sops/code_parser.py:688-696`
- **问题**：推断器被导入并赋值，但返回的 `dataSources` 中无语义描述，功能实现了一半。
- **修法**：在 `parse_code_with_sources()` 中调用推断器并将描述写入 `DataSource`。

---

### M5 · `execution_logs.json` 有脏数据

- **位置**：`data/execution_logs.json:7`
- **问题**：`input_files` 中存在字符串 `"null"`，是上传失败时路径拼接错误写入的。
- **修法**：只在文件成功保存后才追加路径；清理现有脏数据。

---

### M6 · AIPanel 错误提示硬编码

- **位置**：`src/ui/components/layout/AIPanel.vue:34-39`
- **问题**：`catch` 块显示固定字符串，不管实际错误原因是什么。
- **修法**：捕获 `error` 对象并显示动态错误信息。

---

### M7 · `UploadDialog` 用 `alert()` 显示错误

- **位置**：`src/ui/components/knowledge/UploadDialog.vue:46-54`
- **修法**：改用项目已有的 Toast/Notification 组件。

---

### M8 · 并发竞态：读-改-写非原子

- **位置**：`backend/sops/storage.py:29-77`
- **问题**：`get_all_sops()` 和 `save_sop()` 各自独立读文件，两次读之间可能被另一个进程修改。
- **修法**：读-改-写整段加排他锁，或迁移到 SQLite。

## 六、整体评价

**优点**：架构选型合理（Vue 3 + FastAPI + JSON），前后端职责分离清晰，适配器模式处理契约漂移的思路正确，设计 Token 体系完整。

**主要风险**：Windows 平台兼容性（H1）和下载链路断裂（H2）是当前两个阻断性 bug，在修复前项目实际上无法在目标环境正常使用。沙箱安全性（H4）在本地单用户场景可接受，但需在文档中明确标注非安全边界。
